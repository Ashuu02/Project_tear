# Tear — Architecture Reference

> Living document. Update this whenever a significant architectural decision is made.

---

## What Tear Does

Users enter a product name, answer structured questions (Tier 1 + Tier 2), and receive a citation-backed research document and PowerPoint deck — all generated by a multi-agent AI pipeline that runs in real time.

## User Flow (Page by Page)

```
Landing (/)
  → Intake (/intake)              Tier 1: dimensions, goal, depth
  → Tier 2 (/tier2/[sessionId])  Dynamic questions based on Tier 1 answers
  → Context (/context/[sessionId]) Optional: paste notes or upload file
  → Pipeline (/pipeline/[sessionId]) Live SSE agent feed
  → Research (/research/[sessionId]) Interactive 12-section research doc
  → Deck (/deck/[sessionId])      Slide deck viewer + download
       └→ Edit (/deck/[sessionId]/edit)  Canvas editor
       └→ Config (/deck/deck-config/[sessionId]) Theme/layout picker
```

Sessions are identified by a UUID stored in localStorage. No auth in Phase 1.

## Agent Architecture

The pipeline is orchestrated by **Groq LLaMA 3.3 70B** which uses AI SDK tool calls to coordinate three specialist agents in sequence.

```
Orchestrator (Groq LLaMA 3.3 70B)
  ├── Tool: validate_product → Question Agent (Claude Haiku)
  ├── Tool: search_web       → Crawler Agent (Claude + Anthropic web search)
  └── Tool: synthesize_document → Document Agent (Claude Sonnet 4.6)
```

### Agent Details

| Agent | Model | Role |
|---|---|---|
| Question Agent | Claude Haiku | Validates product name, generates Tier 2 questions |
| Crawler Agent | Claude Sonnet | Web research via Anthropic's built-in web search tool |
| Document Agent | Claude Sonnet 4.6 | Synthesizes 12-section research doc from crawl data |
| Hallucination Check | Claude Haiku | Confidence scoring on freshly generated sections |
| Deck Agent | Claude Haiku | Generates slide deck structure from research doc |
| Helper Chat Agent | Claude Sonnet | Always-on chat companion with session context |

### SSE Event Types (stream API → frontend)

| Type | Payload | Purpose |
|---|---|---|
| `agent` | `{ agent, status, message, progress }` | Agent status updates |
| `crawl` | `{ url, message, findings }` | Individual web crawl results |
| `sources` | `{ crawled, total }` | Crawl progress counter |
| `preview` | `{ text }` | Executive summary preview |
| `done` | `{ document }` | Final research doc |
| `error` | `{ message }` | Pipeline failure |

## Research Cache

To avoid regenerating identical research, sections are cached in Supabase by `product_key` (normalized product name).

- **Full cache hit** → orchestrator skipped entirely, all sections served from cache
- **Partial cache hit** → orchestrator only runs for missing sections, cached sections merged in
- Cache uses a refresh-score system (reuse count + age) to decide expiry
- Token usage tracked per operation: `generate` vs `cache_reuse`

## Analytics Stack (added July 2026)

Three tools running in parallel, each serving a different purpose.

### Vercel Analytics
- **What:** Page views, unique visitors, Web Vitals, countries, devices
- **How:** `<Analytics />` component in root layout — zero config
- **Where to check:** Vercel dashboard → your project → Analytics tab
- **Works:** Production only (not localhost)

### PostHog
- **What:** Full funnel tracking + PPTX download rate
- **How:** `PHProvider` wraps root layout, `track()` utility in `src/lib/posthog.ts`
- **Where to check:** posthog.com → Activity → Live events (real time) or Product Analytics → Funnels
- **Works:** localhost + production

#### Events Tracked

| Event | Fired From | Properties |
|---|---|---|
| `product_selected` | Landing (HeroSection) | product_name, model |
| `tier1_started` | Intake page mount | product_name |
| `tier1_completed` | Intake last step submit | product_name, goal, depth, dimensions |
| `tier2_started` | Tier 2 page mount | product_name, question_count |
| `tier2_completed` | Tier 2 last question submit | product_name, question_count |
| `context_reached` | Context page mount | product_name |
| `pipeline_initiated` | Context — generate clicked | product_name, skipped_context, has_text, has_file |
| `pipeline_started` | Pipeline — SSE opens | product_name, model, depth |
| `pipeline_completed` | Pipeline — done event | product_name, sources_count |
| `pipeline_failed` | Pipeline — error event | product_name, error_message |
| `pipeline_stopped` | Pipeline — user stops | product_name |
| `research_viewed` | Research page mount | product_name, sections_count |
| `deck_viewed` | Deck page mount | product_name |
| `pptx_download_initiated` | downloadPptx() called | product_name, session_id |
| `pptx_download_completed` | downloadPptx() success | product_name, session_id |

### Supabase (analytics_events table)
- **What:** Server-side pipeline health — completion rates, section counts, cache reuse, errors
- **How:** `logPipelineEvent()` in `src/lib/analytics.ts`, called from `src/app/api/stream/[sessionId]/route.ts`
- **Where to check:** Supabase dashboard → SQL Editor → query `analytics_events` table
- **Works:** localhost + production (requires real pipeline run, not demo mode)
